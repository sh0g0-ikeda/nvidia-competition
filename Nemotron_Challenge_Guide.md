## 01　このコンペは何をするものか

一言で言うと「NVIDIAのAIに難しい問題をより正確に解けるよう賢くして、正答率を競う」コンペです。

### わかりやすく説明

NVIDIAが作った **Nemotron 3 Nano** というAI（大規模言語モデル）があります。このAIは数学・コード・論理問題などの「推論問題」を解く能力を持っていますが、完璧ではありません。

コンペでは、このAIに出題された問題を解かせて **正答率（Accuracy）** を競います。あなたの仕事は「このAIをより賢くする工夫」をして、問題をより多く正解させることです。

### 提出の流れ

1. Kaggle上でPython notebookを動かす
2. Nemotron Nanoに問題を解かせる
3. その答えをCSVファイルで提出する
4. KaggleがスコアをLeaderboardに反映する

### スコアの仕組み

| 項目 | 内容 |
|------|------|
| 評価指標 | 正答率（Accuracy）＝ 正解数 ÷ 全問題数 |
| 問題のジャンル | 数学・論理・コード・科学推論 |
| モデルの制約 | Nemotron 3 Nano系のみ使用可。別モデルへの置き換えは原則NG |
| 工夫できる範囲 | プロンプト・ファインチューニング・推論方法など |

現在662チームが参加しており、スコア分布は以下の通りです。

## 02　実装の全体像と必須 vs 差別化ポイント

実装は4つのレイヤーで構成されます。下から順に積み上げる構造で、上のレイヤーほど差別化効果が高いです。

```
Layer 4：GRPO（強化学習）         ← 銀〜金の差別化
Layer 3：合成データ生成（SDG）    ← 銀メダルの鍵
Layer 2：SFT（ファインチューニング）← 銅メダル必須
Layer 1：ベースライン構築         ← 全員の前提
```

---

### Layer 1：ベースライン構築　（目標：1〜2日で完成）

**やること：**

1. Kaggle Team Mergeを完了させる（最優先。締切に注意）
2. コンペのデータをDLして構造を理解する
3. vLLMでNemotron 3 Nano（4B or 30B-A3B）を起動する
4. プロンプトテンプレートを作り推論ループを書く
5. 答えを抽出してCSVにまとめ、初提出する

**ポイント：** Nemotronは `enable_thinking` フラグでreasoning on/offを切り替えられます。難しい問題にはon（`thinking=True`）にして、思考トレース（`<think>...</think>`）を出力させてから最終回答を抽出します。

---

### Layer 2：SFT LoRAファインチューニング

| 設定項目 | 推奨値・内容 |
|----------|------------|
| フレームワーク | Unsloth（高速・VRAM節約）または HuggingFace TRL |
| LoRAパラメータ | rank=16〜64、alpha=32〜128、対象: q_proj, v_proj等 |
| バッチサイズ | gradient_accumulation=4〜8でVRAM節約 |
| 学習率 | 2e-4〜5e-5（コサインスケジュール推奨） |
| エポック数 | 1〜3（過学習注意。早期停止で管理） |
| 推定時間 | データ1万件 × 1エポック = T4 GPUで約6〜12時間 |

---

### Test-Time Compute（推論時の工夫）

モデルを変えずにスコアを上げられる手法。SFTと並行して試せます。

| 手法 | 内容 | 目標 |
|------|------|------|
| Majority Voting (maj@N) | 同じ問題をN回推論（N=8〜64）し、最多回答を採用。実装が簡単で効果大 | 銅 |
| Best-of-N Sampling | N回推論して最良の回答を選ぶ。Reward Modelが必要 | 銀 |
| Reasoning Trace制御 | systemプロンプトでthinkingの深さを制御する | 銀 |

---

### Layer 3：合成データ生成（SDG）

→ 詳細は [03章](#03-データ収集合成データ生成深掘り) で説明。

---

### Layer 4：GRPO強化学習（銀〜金メダル向け）

> ⚠️ **前提条件：** SFTで動くモデルがある、GPU時間が週30h以上ある、スコア検証環境がある。この3条件が揃ってから取り組むこと。

SFTで学んだモデルに対し、正解すれば正の報酬・不正解はペナルティを与えながら訓練します。NVIDIAのNeMo RLまたはHuggingFace TRLライブラリを使います。

**GRPOの失敗パターン：**

- **技術的に動かない：** 報酬関数（答えが合っているか判定するスクリプト）の実装で詰まる。答えの形式の揺れ（`3/4` vs `0.75` vs LaTeX記法）への対応が意外と厄介
- **動いたけどスコアが下がる：** 学習率が高すぎるとSFTで学んだ能力を忘れる（catastrophic forgetting）、KL係数が小さすぎると元のモデルから暴走する

**推奨戦略：**

```
SFTモデルA（保険・触らない）  ← 提出用に確保
SFTモデルB → GRPO実験1
            → 結果を分析してから
SFTモデルC → GRPO実験2（Bの結果を踏まえて改善）
```

- GRPOは**直列で**試す（BとCを同時並行にしない）
- 最大2個まで。3個以上は管理コストとGPU時間が足りない
- SFTモデルのweightをコピーすれば学習は1回で済む

## 03　データ収集・合成データ生成（深掘り）

データの質と量が最終スコアを決定します。

### データソース一覧（優先度順）

| データソース | 内容 | 入手方法 | 優先度 |
|------------|------|---------|--------|
| コンペのTrainデータ | コンペが提供するベンチマーク問題と正解ラベル。最重視する | KaggleのDataタブからDL | 最優先 |
| Nemotron-Post-Training-v3 | NVIDIA公式。SFT・RL用3200万+件。reasoning trace付き | `nvidia/Nemotron-Post-Training-Dataset-v3` | 最優先 |
| Llama-Nemotron Post-Training Dataset | 3200万件超。SFT・RL両対応 | `nvidia/Llama-Nemotron-Post-Training-Dataset` | 高 |
| OpenMathReasoning Dataset | 数学特化。AIMO-2金メダルに使ったデータ | `nvidia/OpenMathReasoning` | 数学重視なら高 |
| 合成データ（自作） | 大きなモデルを教師として問題とreasoning traceを自動生成 | 後述の手順で自作 | 銀〜金に必須 |

---

### 公式データのキュレーション手順（5ステップ）

Nemotron-Post-Training-v3は巨大すぎてそのまま使えません。コンペのタスクに合ったサブセットを絞ります。

**Step 1：カテゴリフィルタ**  
コンペのタスクが数学寄り・コード寄り・一般推論かを分析し、対応するカテゴリのみ残す。  
例: `math` / `science` / `reasoning` カテゴリを優先。

**Step 2：トークン長フィルタ**  
`max_completion_token_count = 8192` などの上限を設定し、異常に長いreasoning traceを除外する。  
VRAMに乗りきらない長いサンプルは学習時クラッシュの原因になる。

**Step 3：品質フィルタ（繰り返し除去）**  
reasoning trace内で同じフレーズが繰り返されているサンプルを除外する。  
スライディングウィンドウでN-gramの重複率が閾値を超えたら削除（NVIDIAが公式に採用している手法）。

**Step 4：言語フィルタ**  
コンペが英語問題なら英語サンプルのみ残す。`langdetect` ライブラリで自動分類できる。

**Step 5：サイズ調整**  
最終的に5,000〜30,000件が目安（GPUクォータと相談）。

---

### 合成データ生成（SDG）の具体手順

これが銀〜金メダルへの最大の差別化ポイントです。

**SDGパイプラインの全体像：**
```
コンペ問題
  → シードとして使用
  → 先生モデルに類似問題を生成させる
  → 先生モデルに答えとreasoning traceを生成させる
  → フィルタリング（品質チェック）
  → SFTの訓練データに追加
```

---

**Step 1：先生モデルの選択**

| 選択肢 | コスト | 精度 |
|--------|--------|------|
| Qwen3-235B-A22B | HuggingFace無料枠 | 高 |
| DeepSeek-R1 | APIコストあり | 非常に高 |
| Nemotron 3 Super 120B | NVIDIAのAPIで試用可 | 高 |
| Qwen3-30B（ローカル） | Kaggle GPU枠で動く | 中 |

**Step 2：シードデータの準備**  
コンペのtrain dataに含まれる問題をシードとして使う。問題のテーマ・難易度を分析して多様性を確保する。  
例：数学なら「代数」「組合せ」「幾何」それぞれからシードを取る。

**Step 3：問題生成プロンプト**
```
system: "あなたは数学の出題者です。"
user: "以下の問題と同じ難易度・ジャンルで、異なる問題を5つ生成してください。"
```

**Step 4：Reasoning Trace付き回答生成**
```
system: "あなたは優秀な数学の解法チューターです。"
user: "以下の問題を<think>タグ内でステップごとに考え、最後に<answer>タグで答えを出してください。"

→ モデルが <think>...</think><answer>...</answer> 形式で返す
```

**Step 5：品質フィルタ**
- 答えが形式通りか（answerタグがあるか）
- reasoning traceの長さが適切か（短すぎ・長すぎを除外）
- 答えが複数生成で一致するか（self-consistency check）
- 繰り返しフレーズがないか

**Step 6：データ量の目安**

| 目標 | データ量 |
|------|---------|
| 銅メダル | 公式データ + SDG 5,000〜10,000件 |
| 銀メダル | 公式データ + SDG 20,000〜50,000件 |
| 金メダル | 公式データ + SDG 50,000件以上 + 品質重視 |

## 04　人間が必ずやること（具体的内容）

### ① Team Mergeの実施　【最優先】

KaggleページのTeamタブから友人を招待してチームを結成する。  
**Team Merge Deadlineを過ぎると個人参加のままになる。参加直後に必ず実施。**  
頻度：コンペ参加直後に1回

---

### ② Kaggle Discussionの毎日チェック　【必須】

他の参加者が何を試したか・公開ノートブックに何があるかを把握する。  
- コンペ開始直後に必ずスターターノートブックが公開される。これをforkして改良するのが最速の銅メダル戦略
- 「LB shakeup」（Public/Private LBの乖離）の議論が出たら要注意。提出選択を見直す  

頻度：毎日10〜20分

---

### ③ スコアの解釈とエラー分析　【必須】

提出スコアを見て「どの問題で間違えているか」を分析する。

- 「答えは正しいが形式が違う」→ プロンプト改善で解決
- 「reasoning traceは正しいが最終回答が違う」→ 抽出ロジックの問題
- 「完全に間違い」→ データ追加またはモデル改善が必要
- Public LBが上がっても「本当の実力向上か、Public過学習か」を見極める

頻度：提出のたびに30分〜1時間

---

### ④ 実験の仮説立てと優先度決め　【必須】

「次に何を試すか」を2人で議論して決める。  
優先度は「確実に上がりそうか」「実装コストが低いか」の2軸で評価する。  

頻度：週2〜3回、各30〜60分

---

### ⑤ 最終提出2件の選択　【最重要】

Kaggleでは最終順位を決めるPrivate LBに使う提出を2件選べる。デフォルトはlast 2 submissionsだが**必ず手動で選ぶこと。**

- スコアが高いもの ＋ 過学習していなさそうなものを選ぶ
- 2件に多様性を持たせる（SFT版 + SFT+GRPO版など）のがリスクヘッジ
- **締切前日に2人で必ず確認する**

頻度：締切前日・当日に集中的に

---

### ⑥ GitHub・実験ログ管理　【必須】

**推奨リポジトリ構成：**
```
notebooks/      ← Kaggle提出用
experiments/    ← 実験記録（スコアとパラメータをMarkdownで記録）
data/           ← データ処理スクリプト
scripts/        ← SFT・推論スクリプト
```

- コミットごとにスコアをコメントに残す
- 毎日夜に実験結果をSlack/LINEで2人共有するルールを決める

頻度：毎日

---

### ⑦ GPU・計算資源の管理　【必須】

- 1人あたり週30時間 × 2人 = 60時間が上限
- 大きな実験（SFT 12時間等）はクォータが余っている週初めに回す
- デバッグは小さいデータ（100件）で素早くやる
- 必要に応じてRunPod（H100 約$3〜4/時間）を使う

頻度：週1回の計画確認

---

### ⑧ SDG生成データの品質手動確認　【銀〜金メダル向け】

自動生成した合成データを50〜100件ランダムサンプリングして手で読む。

**チェックポイント：**
- reasoning traceが論理的に正しいか
- 答えが問題と対応しているか
- 言語・形式が一貫しているか

品質が低いバッチは丸ごと除外する判断をする。スクリプトだけでは検出できないノイズがある。

頻度：データ生成のたびに1〜2時間

## 07　LLMエージェントに任せられる範囲

### エージェントに丸投げできるタスク

| タスク | 具体的にできること |
|--------|-----------------|
| 推論ループのコード生成 | 「vLLMでNemotronを動かして問題を解かせるPythonスクリプトを書いて」で一発で出る |
| SFTスクリプトのボイラープレート | 「UnslothでLoRA SFTするコードを書いて。データはJSONL形式。モデルはNemotron 4B。」で動くコードが出る |
| データフィルタリングスクリプト | 繰り返し検出・トークン長フィルタ・言語フィルタ等のコード生成 |
| SDGのプロンプト設計 | 「数学問題を生成するシステムプロンプトを書いて」で複数案が出る |
| エラーデバッグ | トレースバックを貼り付けるだけで原因と修正案が返ってくる |
| Kaggle提出用CSVの整形 | 「このデータフレームをこの形式のCSVに変換して」で完結 |
| 論文・ドキュメントの要約 | NeMo Skills・TRL・Unslothのドキュメントを貼り付けて「使い方を教えて」でOK |
| 実験記録のMarkdown整形 | 「このスコアと設定をMarkdownの表にして」で実験ログを整理 |

### 人間が必ず介在しないといけないタスク

| タスク | 理由 |
|--------|------|
| コンペ状況の把握（Discussion読解） | 「今週みんな何を試しているか」はエージェントにはわからない |
| スコアが上がった理由の判断 | Public LBが上がっても「本物の改善か・過学習か」の判断は文脈が必要 |
| 実験の優先順位決め | 時間・GPU・スコアを総合した意思決定はエージェントに委ねられない |
| 最終提出2件選択 | リスクとリターンのバランスを判断する。コンペで最も重要な意思決定 |
| SDGデータの品質手動確認 | 生成データを実際に読んで「これは使えるか」を判断する作業はスクリプトで代替できない |
| チームの状況管理 | どこを諦めてどこに集中するかの判断はエージェントにできない |

## 08　銅 vs 銀メダル チェックリスト

### 銅メダルを取るためのチェックリスト

- [ ] Kaggle Team Mergeが完了している
- [ ] Nemotron 3 Nanoで推論ループが動き、初回提出ができている
- [ ] Majority Voting（N≥8）が実装されている
- [ ] Nemotron-Post-Training-v3から5,000件以上のサブセットを用意できている
- [ ] LoRA SFTが1回以上実施され、ベースラインよりスコアが上がっている
- [ ] 提出のたびにエラー分析を行い、改善ポイントを記録している
- [ ] 最終提出2件の選択を締切前日までに確定している
- [ ] GitHubで実験ログが管理されている

### 銀メダルを取るためのチェックリスト（銅に追加）

- [ ] SDGパイプラインが構築され、20,000件以上の合成データが生成できている
- [ ] 合成データの品質フィルタ（繰り返し除去・self-consistency check）が実装されている
- [ ] 合成データを使ったSFTでスコアが銅メダルレベルから有意に改善している
- [ ] Majority Voting N=64以上が実施できている
- [ ] GRPOの実装を試みており、SFTよりスコアが上がっている
- [ ] SDG生成データを人間が50件以上サンプリングして品質確認している
- [ ] Public LBが銀メダルラインを超えており、Private LBとの乖離も考慮した提出選択をしている

### 金メダルを狙うためのチェックリスト（銀に追加）

- [ ] SDGデータが50,000件以上あり品質が高い
- [ ] GRPOがSFTから+0.03以上の改善をもたらしている
- [ ] 報酬関数が精緻に設計されており、崩壊していないことを確認している
- [ ] SFTモデルAを保険として確保した上でGRPO実験2本を直列で実施している
- [ ] スコアの推移を見て終盤のshakeupリスクを考慮した提出選択をしている
